## 2026-06-24 — Document `keitaro_stage_results.cost` is always 0 (do-not-use guard) — docs: 03-data-model, CHANGELOG
- **Why.** `keitaro_stage_results.cost` is Keitaro ad-platform spend, which is always 0 here (we don't buy traffic through Keitaro). Any ROI/EPC/profit built off it would be wrong (e.g. infinite ROI from cost 0). Real stage cost is the SMS send spend in `campaign_stages.total_cost`.
- **Audit.** Confirmed no ROI or EPC anywhere derives cost from this column: the campaign-detail ROI tiles + per-stage ROI use `campaign_stages.total_cost`; the manual-results ROI estimate uses `total_cost`; the `/reports` route (`app/api/keitaro/reports`) folds this column in but **overwrites** it with `campaign_stages.total_cost` before computing profit; stored/derived `epc` everywhere is `revenue / clicks` (revenue-derived, not cost). The unused `/api/keitaro/results` endpoint exposes a Keitaro-cost-based `profit` but has no caller.
- **Change.** Added a `DO NOT USE` comment to the `cost` column (and a clarifying note to `epc`) in [db/schema.ts](../db/schema.ts) and flagged it in the data-model table. No schema/migration change, no behavior change.

## 2026-06-24 — Standardize all sales/revenue reporting on conversion date (`stat_date`) — docs: 07-conventions, 03-data-model, 04/keitaro-poll, CHANGELOG
- **Problem.** Sales/revenue were counted two ways that silently disagreed: the `/reports` page bucketed by Keitaro conversion day (`keitaro_stage_results.stat_date` — e.g. 06-23 = 10 sales / $741) while the dashboard stats + daily-activity charts bucketed by the stage's **send day** (06-23 = 8 sales / $600). Conversions lag sends, so the two windows differ, and nothing defined which to use.
- **Fix.** New single source of truth [lib/reporting/attribution.ts](../lib/reporting/attribution.ts) with `ATTRIBUTION_BASIS = 'stat_date'` and shared helpers (`salesRevenueTotals`, `salesRevenueByDay`, `manualSalesByStageInRange`). Revenue + Keitaro sales date by `stat_date`; manual sales date by the `stage_manual_sales` ledger **entry date**; the two sales sources combine per stage via `combineSales` (max). Rewired [app/api/dashboard/stats/route.ts](../app/api/dashboard/stats/route.ts), [app/api/dashboard/daily-activity/route.ts](../app/api/dashboard/daily-activity/route.ts), and [app/api/keitaro/reports/route.ts](../app/api/keitaro/reports/route.ts) (manual baseline moved from `sent_at` to the ledger). Dashboard now shows 06-23 = 10 sales / $741, matching `/reports`. Send-day metrics (SMS, cost, opt-outs, clickers) stay send-dated; a code note marks that daily ROI mixes windows (revenue conversion-dated, spend send-dated — washes out over a steady period).
- **Migration 0084** (`0084_redate_manual_sales_backfill.sql`): re-dates the migration-0079 backfill ledger rows (`entered_by IS NULL`, all stamped at one `now()`) to each stage's effective send day, so historical manual sales spread realistically across the calendar instead of lumping on a single date. Data-only, idempotent. **Must be applied via `npm run db:migrate` before/at deploy** — until then manual sales read as lumped on the backfill date.
- **Optional dual view** ("booked today" stat_date vs "maturing" send-day cohort side-by-side) flagged but NOT built — deferred as non-trivial.

No-op for healthy campaigns (visit rows normally carry no conversions). No schema change.
- **Test.** [scripts/test-keitaro-visit-conversions.ts](../scripts/test-keitaro-visit-conversions.ts) — pure unit test of the fold (18 assertions): visit-row conversions credited, clicks split correctly, combined fold sums without double-count, healthy visit row unchanged.

Exactly **one** `opt_out_attributions` row + one stage's `inbound_opt_out_count`/`opt_out_count`/`total_cost` bump per STOP. Unattributed + idempotency (`texthub_inbound_events` claim + `ON CONFLICT`) unchanged; `attributed` summary is now 0 or 1 per STOP. No schema change — the existing partial index `stage_sends_org_phone_sent_idx` already serves the global-latest lookup.
- **CSV imports unaffected:** the result-import path ([app/api/campaigns/[campaignId]/stages/[stageId]/import/route.ts](<../app/api/campaigns/[campaignId]/stages/[stageId]/import/route.ts>)) never writes `opt_out_attributions` — it bumps `opt_out_count` for the one stage being imported — so it already attributes to a single stage. Left as-is.
- **Backfill.** [scripts/backfill-optout-latest-stage.ts](../scripts/backfill-optout-latest-stage.ts) collapses historical fan-out (keep latest-stage row per opt_out, re-derive each affected stage's counters + `total_cost`). Dry-run by default, `--apply` to commit; idempotent. Dry-run on the shared DB: 4191 → 3014 attribution rows (= distinct opt_outs), 1177 collapsed, 21 stages. **Not run with `--apply` yet.**
- **Tests.** [scripts/test-optout-latest-stage.ts](../scripts/test-optout-latest-stage.ts) (rolled-back, real schema): 3-stages→latest, tie-break, unattributed, idempotent re-credit — all green. Existing `verify-poll-opt-outs.ts` harness untouched + green.

## 2026-06-21 — Reports: Sales = max(manual, Keitaro) not sum (fix double-count) + Avg Opt-out badge — docs: 04/keitaro-poll, CHANGELOG
- **Bug — Sales double-counted a sale present in both Keitaro and the manual tally.** Sales was **additive** (`manual sales_count + Keitaro conversions`). Applied to both `/reports` ([app/api/keitaro/reports/route.ts](../app/api/keitaro/reports/route.ts), `max(0, manual − Keitaro)` top-up in range) and the campaign-detail page's 3 sales sites ([app/(protected)/campaigns/[id]/page.tsx](../app/(protected)/campaigns/[id]/page.tsx)). No data migration; read-time math only. The dashboard (manual `sales_count` only, never additive) is unaffected.
- **Feature — Avg Opt-out stat card on `/reports`.** New grand-total badge showing the period's aggregate opt-out rate (grand opt-outs ÷ grand total sent), alongside the existing Clickers/Offer Redirect/Sales/Revenue/Cost/Profit cards ([app/(protected)/reports/page.tsx](../app/(protected)/reports/page.tsx)). ## 2026-06-20 — Reports: fix Total Sent for manual sends + anchor manual sales to the send date — docs: 04/keitaro-poll, CHANGELOG
- **Bug 1 — Total Sent showed 0 for manually-sent stages.** `/api/keitaro/reports` counted only `stage_sends` (the TextHub API pipeline). Almost all sends happen manually in external provider tools, which have no `stage_sends` rows, so every manual stage read 0. The `stage_manual_sales` ledger remains for audit/current-total but no longer drives the report. No data migration needed.

## 2026-06-19 — Spam check: stop failing on new creatives (cold-start hardening) — docs: 04/spam-classifier, CHANGELOG
- **Bug:** scoring a *new* creative (cache miss → real classifier call) often failed because the Cloud Run classifier scales to zero and the first request's cold start blew the 10s timeout. The fallback (score 50) was then **cached permanently**, pinning the text to a bogus "50 / NOT SPAM" even after the service recovered.
- Per-attempt timeout 10s → **25s** (`CLASSIFIER_TIMEOUT_MS`); provider **retries once** on timeout/network errors only (not on HTTP non-2xx); `/api/spam/score` gains `maxDuration = 60` ([lib/spam/providers/classifier.ts](../lib/spam/providers/classifier.ts), [app/api/spam/score/route.ts](../app/api/spam/score/route.ts)).
- **Failures are no longer cached** ([lib/spam/score.ts](../lib/spam/score.ts)) — a transient failure no longer poisons the cache; the next check re-scores.
- Strip shows **"Couldn't score — try Re-check"** instead of a fake verdict on failure ([components/spam/spam-check-strip.tsx](../components/spam/spam-check-strip.tsx)).
- Recommended permanent fix (ops, outside repo): run the classifier always-on (self-hosted / `min-instances=1`) — one-env-var change app-side (`CLASSIFIER_URL`).

## 2026-06-20 — Reports: every metric scoped to the selected date range + dated manual-sales ledger — migration 0079 — docs: 03-data-model, 04/keitaro-poll, CHANGELOG

  - Range → UTC bounds via `fromZonedTime` (America/New_York); exclusive upper bound built off the next calendar day (not +24h) so it's DST-correct. Opt-outs / total-sent / manual-sales each computed by one grouped query over the stages in view; all summed into campaign rollups + grand totals, both new fields added to the sortable set + response `totals`.
- **Ledger writes:** the manual-results endpoint ([app/api/campaigns/[campaignId]/stages/[stageId]/manual-results/route.ts](../app/api/campaigns/[campaignId]/stages/[stageId]/manual-results/route.ts)) now records `delta = new − old sales_count` (when non-zero) into `stage_manual_sales` in the same transaction as the stage update, stamped with the actor. Migration 0079 backfills existing `sales_count > 0` totals as one delta dated `now()` ("treat as entered now").
- Page ([app/(protected)/reports/page.tsx](../app/(protected)/reports/page.tsx)) renders the **Total Sent** + **OptOut, %** columns. Schema: [db/schema.ts](../db/schema.ts) `stage_manual_sales`. Typecheck clean. **Migration 0079 must be applied (`npm run db:migrate`) before deploy.**

## 2026-06-12 — Fix: scheduled tracked sends timed out + got stranded (full pipeline fix)
- **Root cause of "scheduled send never fired":** the cron materialized + sent a whole audience inside one 300s invocation, minting links **one-by-one** (~178s for 1000 recipients) AFTER committing the `sent_at` claim — so a timeout left the stage `sent_at`-locked with zero rows, and the scheduler never retried it. Three-part fix:
  1. **Batched minting** ([lib/links/mint-link.ts](../lib/links/mint-link.ts) `mintLinksBatch` + [lib/sends/kickoff.ts](../lib/sends/kickoff.ts) bulk inserts): single shared destination upsert + multi-row chunked `links`/`stage_sends` inserts. 1000-recipient kickoff ~178s → ~2-3s (measured).
  2. **Claim-after-materialize** ([lib/sends/scheduled.ts](../lib/sends/scheduled.ts)): `sent_at` is now stamped only AFTER kickoff succeeds; concurrency relies on the `(stage_id, contact_id)` active-send dedup index. A killed tick no longer strands the stage.
  3. **Resumable drain across ticks:** phase B (`selectDrainableStages`) drains any tracked stage with `pending` rows in budget-bounded batches, independent of `sent_at` — large audiences send across multiple `*/15` ticks instead of one 300s window.
  - Tests: rewrote `test-scheduled-budget`/`test-scheduled-run` to the two-phase counters; added `test-scheduled-e2e` (two-tick materialize→drain→resume, rolled back) and `verify-batch-kickoff` (timing/correctness); made `verify-send` fixture self-contained. — Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).

## 2026-06-11 — Fix: manual "mark sent" silently cancelled scheduled tracked sends
- `campaign_stages.sent_at` is the `send-scheduled` cron's fire-lock (`selectDueScheduledStages` requires `sent_at IS NULL`). The legacy status action ([…/stages/[stageId]/status/route.ts](../app/api/campaigns/[campaignId]/stages/[stageId]/status/route.ts)) stamped the same column when moving a stage to `'sent'`, so marking a tracked stage `'sent'` before its scheduled time stamped `sent_at` and permanently removed it from the scheduler — and reverting the status didn't clear it (real incident: a 3 PM ET send never fired). Now the status route **rejects `'sent'` for tracked campaigns** (409 `tracked_stage_sent_is_pipeline_owned`), and the stage status dropdown hides `'sent'` for tracked campaigns ([campaigns/[id]/page.tsx](../app/(protected)/campaigns/[id]/page.tsx)). Added read-only `scripts/diagnose-scheduled.ts`. — Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).
