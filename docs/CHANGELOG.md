# Docs Changelog

A running log of documentation-affecting changes. Add a dated entry whenever a doc is materially updated, and note the code commit/migration that prompted it.

## 2026-07-07 — Telegram hourly report: fix silent cold-start hang + build-phase blind spot — docs: 04-features/crons, CHANGELOG
- **Symptom:** the hourly performance report stopped arriving for days while the **daily** report kept coming — and no `⚠️ failed` alert fired either (total silence). **Root cause (two compounding).** (1) `buildHourly` ([app/api/cron/telegram-report/route.ts](../app/api/cron/telegram-report/route.ts)) ran `2× computeReportMetrics` via `Promise.all` = **8 concurrent queries**; on a cold serverless start during busy ET hours the concurrent connection acquisition stalled the pooler past `maxDuration=60` and the function was killed. Queries are ~16 ms each — the cost was cold concurrent *connections*, not execution — so it only bit the heavier hourly path (daily fires 4 queries at the quiet 05:00 ET hour and never tripped it). (2) The metrics **build ran outside the send try/catch**, so that kill produced no report *and* no alert. Diagnosed live: a plain hourly call hung 40 s+ (vs 1.9 s warm) while the DB sat idle and `salesRevenueTotals` EXPLAIN-ANALYZE'd at 16 ms.
- **Fix.** Hourly now fetches today's full metrics (4 queries) + yesterday's **spend only** (`spendInRange`, newly exported from [lib/reporting/report-snapshot.ts](../lib/reporting/report-snapshot.ts)) **sequentially** — 8→5 queries, peak concurrency 8→4 (matches daily). Build + send are wrapped in one try/catch under a **50 s `withTimeout`** (below `maxDuration`), so a hung build now returns 500 **and** fires the `notifyTelegram` `⚠️` alert instead of dying silently. Builds on the 2026-07-03 retry/alert hardening (`d6abfe8`).

## 2026-07-07 — Fix "Could not preview audience" on the stage form (timeout mis-detection) — docs: 04-features/content-dedup, CHANGELOG
- **Symptom:** the stage form's STAGE AUDIENCE block failed entirely with "Could not preview audience.", blocking the whole panel. **Root cause:** `computeStageEligibilityPreview` ([lib/audience-snapshot.ts](../lib/audience-snapshot.ts)) is meant to catch a `SET LOCAL statement_timeout` cancellation (SQLSTATE `57014`) and degrade to a `truncated` preview, but it checked `err.message` — and postgres-js via drizzle puts the `57014` code and "statement timeout" text on `err.cause`, not `err.message` (the message is `"Failed query: …"`). So a legitimate timeout re-threw → the route 500'd → even the working audience count (which the UI already renders alongside a "too large to compute the dedup preview" note) was lost. Surfaced "recently" because the eligibility query only crosses the 10s limit intermittently (cold serverless connections + grown exposure/opt-out ledgers). Reproduced against a real campaign: a cold-connection run cancelled with `57014` and re-threw.
- **Fix:** new [`isStatementTimeout`](../lib/db/statement-timeout.ts) walks the error `cause` chain (code `57014` first, message fallback, cycle-guarded); wired into both `computeStageEligibilityPreview` and `segmentRulesPreview` ([lib/segment-rules-eval.ts](../lib/segment-rules-eval.ts)), which shared the same latent bug. Now a timeout degrades to `truncated` (block shows the count + "too large to compute the dedup preview — it still applies at send time") instead of 500'ing. Verified: unit test `scripts/test-statement-timeout.ts` (drizzle-wrapped/direct/message/nested/cyclic/negative shapes) + a forced 1ms-timeout run of the real function returning `truncated` without throwing. No schema change.

## 2026-07-07 — Campaign stages list no longer stalls 30–60s on behavioral lanes — docs: 04-features/behavioral-lanes, 07-conventions
- Opening a campaign with a behavioral split took 30–60s to render Stages. Root cause: each lane's Audience count ran a live-tier scan (`links⋈clicks` + `stage_sends`, ~6.6s cold) inline in `GET /api/campaigns/[campaignId]/stages`, once per lane (3 per split), fired in parallel and CPU-contending. Fix: (1) new `computeLaneAudienceCountsBatch()` ([lib/audience-snapshot.ts](../lib/audience-snapshot.ts)) computes ALL lanes of a campaign in ONE query — the tier map is a single `MATERIALIZED` CTE reused per lane, the parent "alive" set built once; (2) lane counts are deferred off the list — the stages route returns lanes with `audience_count = null` (client renders `computing…`) and the client fetches the new `GET /api/campaigns/[campaignId]/stages/lane-counts` after first paint. List now paints in ~0.5s; lane numbers fill in shortly after. Batched counts proven byte-identical to the old per-lane `countStageRecipients` path across 25 real lane campaigns by [scripts/verify-lane-batch.ts](../scripts/verify-lane-batch.ts). No schema change.

## 2026-07-07 — Stage delete gate now covers behavioral lanes; atomic self-guarding delete — docs: 04-features
- Final whole-branch review of stage hard-delete found the gate only checked the TARGET stage's own send/result tables, so a never-sent behavioral parent could pass the gate and CASCADE-delete its sent lanes' data. `deleteStage` ([lib/stages/delete-stage.ts](../lib/stages/delete-stage.ts)) now blocks (409 `stage_has_send_data`) when the target OR any `parent_stage_id`-lane has `sent_at` set or rows in `stage_sends`/`stage_results_imports`/`stage_manual_sales`/`keitaro_stage_results`; a parent whose lanes are all truly empty still deletes (undo-split path unaffected). Gate + delete are now one atomic self-guarding `DELETE ... WHERE NOT EXISTS (...)` inside the transaction (was gate-SELECT-then-DELETE, a TOCTOU window for a concurrent Prepare). UI Delete action also now hides on a materialized/Prepared stage. No schema change.

## 2026-07-06 — Opt-out backfill from TextHub inbox CSV export (200-cap scroll-off remediation) — docs: 06-integrations, CHANGELOG
- **Cause.** TextHub's `?inbox=true` returns only the 200 most-recent messages, no pagination. Under STOP-reply bursts the 15-min poller couldn't keep up, so ~30% of opt-outs (Jun 29–Jul 5; 46% on Jul 4) scrolled off before ingest and were never suppressed — a live compliance gap, and undercounted per-stage opt-out rates.
- **Backfill.** One-time throwaway script (`scripts/`, deleted after) parsed the full inbox CSV export (2026-05-27 → 07-06, 25,710 `YES` rows), deduped against `texthub_inbound_events` by `(phone, received_at)`, and imported the **8,420 missing** opt-outs with byte-for-byte poller semantics: org-wide `opt_outs` (`source=sms_inbound`, real Mountain→UTC timestamp), live-72h attribution (`latestSendForAttribution`) → `opt_out_attributions` + `inbound_opt_out_count`/`opt_out_count` bumps + stage total-cost recompute. Idempotent/resumable via synthetic `backfill:<phone>:<received>` `provider_message_id`. Result: 4,063 attributed across 95 stages, 4,357 org-wide-only. No schema/code change — data + reporting only.
- **Follow-up (same day): attribution-date fix.** The Reports page (`app/api/keitaro/reports/route.ts`) buckets per-stage opt-outs by `opt_out_attributions.created_at`, which the poller (and this backfill) leaves at `now()`. That made all backfilled attributions cluster on the import day. Corrected `opt_out_attributions.created_at = opt_outs.created_at` (the real STOP time) for the 4,298 rows whose ET report-day differed — opt-outs now report on the day they actually happened.
- **Poller fix (`lib/sends/poll-opt-outs.ts`).** Root of the same class of bug: the live poller stamped both `opt_outs.created_at` and `opt_out_attributions.created_at` at **poll time** (`now()`), so a STOP replied just before an ET-midnight poll landed on the wrong report day. Both inserts now use `anchorIso` — the STOP's parsed `received_at` (Mountain→UTC), with the existing `now()` fallback when unparseable — so opt-out and attribution always share the true reply timestamp and the Reports page/opt-out list agree on the day. Verified via `scripts/verify-poll-opt-outs.ts`.

## 2026-07-03 — Split-leak fix (hash bucketing) + global 1-hour send-dedup (migration 0090) — docs: 04-features/sms-send-pipeline, 03-data-model, 07-conventions, CHANGELOG
- **Split leak.** A split stage's bucket was `row_number() over (order by contact_id) % splitTotal`, computed AFTER the resumable "exclude already-materialized" filter. On every resume the row numbers were reassigned over the shrunken remaining set, so the modulo picked a different subset and **leaked the sibling stage's half** into this one. Incident: campaign `8_62_070326_1` materialized 7,500 instead of 5,000 per split half (5,000 contacts in BOTH stages; 4,926 people got two live SMS). Fixed by a **stable per-contact hash bucket** (`splitBucketMatch`, [`lib/sends/split-bucket.ts`](../lib/sends/split-bucket.ts)) — a contact's bucket depends only on its own id, so excluding materialized rows can't move it. Applied to all six sites that reproduced the split (send recipients, all-phones export, frozen/draft/batched audience counts, dedup will_send preview, reconciliation). Test: [`scripts/test-split-resume-leak.ts`](../scripts/test-split-resume-leak.ts).
- **1-hour send-dedup (safety net).** New HARD gate in the drain: a phone that already received a message within `SEND_DEDUP_WINDOW_MS` (1h, [`lib/sends/dedup-window.ts`](../lib/sends/dedup-window.ts)) — org-wide, across every campaign/stage — is marked **`skipped_duplicate`** (migration [`0090`](../db/migrations/0090_stage_sends_skipped_duplicate.sql); terminal, not sent, not opted-out, not auto-retried) instead of being delivered. Telegram alert + in-app warning (Activity summary card, status badge/filter, stage operational status). Test: [`scripts/test-send-dedup.ts`](../scripts/test-send-dedup.ts).

## 2026-07-03 — Windowed, resumable send materialization (migration 0089) — docs: 04-features/sms-send-pipeline, 04-features/daily-volume-ui, 03-data-model, CHANGELOG
- **What.** "Prepare" (kickoff) materialized a stage's whole audience in ONE atomic transaction. Measured ~3.5 ms/recipient (enumerate + mint links + insert stage_sends across heavily-indexed tables), so it hit the 60s function limit around ~17K recipients and **rolled the whole batch back** — nothing prepared. Now materialization is **windowed + resumable**.
- **Core.** New `campaign_stages.materialized_at` (migration [`0089`](../../db/migrations/0089_stage_materialized_at.sql)) — the completeness signal, set only when the last window lands (backfilled for pre-existing stages that already had rows). `kickoffStageSend` commits in 2000-row windows under a time budget, enumerating only not-yet-materialized recipients, idempotent via `ON CONFLICT DO NOTHING` + the enumerate exclusion. A timeout/crash resumes from committed rows; a re-Prepare is a no-op (`already_pending` retired).
- **Never sends partial.** Scheduler Phase A resumes stages with `materialized_at IS NULL`; Phase B drains only `materialized_at IS NOT NULL`. So a half-built audience can't be sent — the cron finishes materialization, then drains.
- **Guardrail (shipped separately).** Kickoff/approve-send `maxDuration` 60→300.
- **Visibility.** A stage mid-materialization reads a new Indigo **"Materializing"** operational status (not "Prepared") so the operator sees steady progress instead of a stalled spinner; flips to Blue "Prepared" when complete. `materialized_at` surfaced on the stages + sends/today endpoints.
- Verified: `scripts/test-resumable-materialization.ts` (13/13 — fresh, idempotent, resume-after-partial with no duplicates, no-early-send gate), `tsc`, production build. Migration applied to prod.

## 2026-07-02 — Fix: prior-offer dedup (LAYER 3) self-cannibalized single-offer drips — docs: 04-features/content-dedup, CHANGELOG
- **What.** The offer-exposure exclusion (`campaigns.exclude_prior_offer_contacts`, eligibility LAYER 3) excluded EVERY contact in `offer_exposures` for the offer, with no same-campaign carve-out — unlike LAYER 1 (creative), which excepts `campaign_id <> current`. The always-on creative layer still blocks re-sending identical content within a campaign.
- **Data.** The one affected stage (667) was recalled (its single stale `pending` row → `rejected`, `send_approved=false`) so it won't send; the operator recreates it. No other campaigns were created after the feature shipped, so none else affected.

## 2026-07-02 — Speed Tier 3: capped contacts-list count (kills the ~670ms every-load count) — docs: 07-conventions, 04-features/contacts-and-groups, CHANGELOG
- **What.** `/api/contacts/list` ran an exact `count(*)` on every load; at 752K rows that's ~670 ms (inherently O(rows) — an index can't help, proven by forcing the index path). Now the count is **capped**: it counts at most `COUNT_CAP+1` (10,000) rows via a `LIMIT` subquery. Under the cap it's the exact total; over it, the response sets `countApprox:true` + `totalCount:10000` and the UI shows "10,000+". **670 ms → 84 ms.**
- **Navigation stays correct.** The count no longer drives paging: the page query fetches `pageSize+1` and returns `hasMore`, so Previous/Next works past the cap. `DataTable` gained optional `hasMore` + `totalCountApprox` props (backward-compatible — other lists pass nothing and keep exact-count/`of Y` behavior). When approximate, the footer shows "N+" and "Page X" (no "of Y").
- **Exact counts preserved when it matters.** Any narrowing filter (search — already trigram-fast per Tier 1, segment, group, or the opt_outs/opt_ins/clickers views) returns a set under the cap, so the count stays exact. Only the unfiltered full-org browse shows "10,000+".
- Verified: capped-count speed + semantics (`scripts/verify-capped-count.ts`), Drizzle subquery SQL runs, `tsc`, production build. Scoped to the contacts endpoint (the measured pain); the `DataTable` mechanism is reusable for other large lists later.

## 2026-07-01 — Speed Tier 2: batched Keitaro pollers, staggered crons, parallelized queries, lazy campaign-detail dialogs — docs: 04-features/crons, CHANGELOG

## 2026-06-12 — Fix: scheduled tracked sends timed out + got stranded (full pipeline fix)
- **Root cause of "scheduled send never fired":** the cron materialized + sent a whole audience inside one 300s invocation, minting links **one-by-one** (~178s for 1000 recipients) AFTER committing the `sent_at` claim — so a timeout left the stage `sent_at`-locked with zero rows, and the scheduler never retried it. Three-part fix:
  1. **Batched minting** ([lib/links/mint-link.ts](../lib/links/mint-link.ts) `mintLinksBatch` + [lib/sends/kickoff.ts](../lib/sends/kickoff.ts) bulk inserts): single shared destination upsert + multi-row chunked `links`/`stage_sends` inserts. 1000-recipient kickoff ~178s → ~2-3s (measured).
  2. **Claim-after-materialize** ([lib/sends/scheduled.ts](../lib/sends/scheduled.ts)): `sent_at` is now stamped only AFTER kickoff succeeds; concurrency relies on the `(stage_id, contact_id)` active-send dedup index. A killed tick no longer strands the stage.
  3. **Resumable drain across ticks:** phase B (`selectDrainableStages`) drains any tracked stage with `pending` rows in budget-bounded batches, independent of `sent_at` — large audiences send across multiple `*/15` ticks instead of one 300s window.
  - Tests: rewrote `test-scheduled-budget`/`test-scheduled-run` to the two-phase counters; added `test-scheduled-e2e` (two-tick materialize→drain→resume, rolled back) and `verify-batch-kickoff` (timing/correctness); made `verify-send` fixture self-contained. — Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).
