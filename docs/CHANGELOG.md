## 2026-06-17 — Behavioral-branching: behavioral-split endpoint (build step 4) — no schema change — docs: CHANGELOG only
- New endpoint `POST /api/campaigns/[campaignId]/stages/[stageId]/behavioral-split` ([route](../app/api/campaigns/[campaignId]/stages/[stageId]/behavioral-split/route.ts)). Turns the chosen stage into a PARENT position (it stays ordinary) and stamps three lane-stages under it — `behavioral_tier` 0/1/2 (ignored/clicked/reached); tier 3 converted gets no lane. Each lane clones the parent's config, sets `behavioral_tier` + `parent_stage_id = chosen stage`, regenerates its own stage tracking_id, and leaves `split_index/split_total` NULL (lanes partition by tier, not the equal-size A/B partition). Transactional — a partial split can't persist. Modeled on the A/B split route's transaction/clone structure.
- Logic factored into [lib/stages/behavioral-split.ts](../lib/stages/behavioral-split.ts) (`performBehavioralSplit`) so it's testable without the auth session; the route is a thin auth (`stages.create`) + error-map wrapper. Guards: reject if the target is itself a lane (`already_lane`), archived, or already behaviorally split (`already_behaviorally_split`). **No draft/status gate** — the A/B split route has none either, and behavioral lanes are created post-activation by design. utm_tag_ids is cloned too (A/B split omits it; included here for config completeness).
- NOT wired into UI/sending yet. Tests: [scripts/test-behavioral-split.ts](../scripts/test-behavioral-split.ts) — 16/16 under a dedicated throwaway org (teardown scoped to org_id, marker-guarded; real-data counts asserted unchanged): 3 lanes with tiers 0/1/2 + correct parent + distinct tracking IDs + NULL split fields; splitting a lane rejected; the step-1 CHECK rejects tier-set/parent-NULL and tier=3; and a forced mid-transaction tracking_id collision rolls back with zero orphan lanes.

## 2026-06-12 — Fix: scheduled tracked sends timed out + got stranded (full pipeline fix)
- **Root cause of "scheduled send never fired":** the cron materialized + sent a whole audience inside one 300s invocation, minting links **one-by-one** (~178s for 1000 recipients) AFTER committing the `sent_at` claim — so a timeout left the stage `sent_at`-locked with zero rows, and the scheduler never retried it. Three-part fix:
  1. **Batched minting** ([lib/links/mint-link.ts](../lib/links/mint-link.ts) `mintLinksBatch` + [lib/sends/kickoff.ts](../lib/sends/kickoff.ts) bulk inserts): single shared destination upsert + multi-row chunked `links`/`stage_sends` inserts. 1000-recipient kickoff ~178s → ~2-3s (measured).
  2. **Claim-after-materialize** ([lib/sends/scheduled.ts](../lib/sends/scheduled.ts)): `sent_at` is now stamped only AFTER kickoff succeeds; concurrency relies on the `(stage_id, contact_id)` active-send dedup index. A killed tick no longer strands the stage.
  3. **Resumable drain across ticks:** phase B (`selectDrainableStages`) drains any tracked stage with `pending` rows in budget-bounded batches, independent of `sent_at` — large audiences send across multiple `*/15` ticks instead of one 300s window.
  - Tests: rewrote `test-scheduled-budget`/`test-scheduled-run` to the two-phase counters; added `test-scheduled-e2e` (two-tick materialize→drain→resume, rolled back) and `verify-batch-kickoff` (timing/correctness); made `verify-send` fixture self-contained. — Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).

## 2026-06-11 — Fix: manual "mark sent" silently cancelled scheduled tracked sends
- `campaign_stages.sent_at` is the `send-scheduled` cron's fire-lock (`selectDueScheduledStages` requires `sent_at IS NULL`). The legacy status action ([…/stages/[stageId]/status/route.ts](../app/api/campaigns/[campaignId]/stages/[stageId]/status/route.ts)) stamped the same column when moving a stage to `'sent'`, so marking a tracked stage `'sent'` before its scheduled time stamped `sent_at` and permanently removed it from the scheduler — and reverting the status didn't clear it (real incident: a 3 PM ET send never fired).
