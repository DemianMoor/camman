## 2026-06-12 ŌĆö Fix: scheduled tracked sends timed out + got stranded (full pipeline fix)
- **Root cause of "scheduled send never fired":** the cron materialized + sent a whole audience inside one 300s invocation, minting links **one-by-one** (~178s for 1000 recipients) AFTER committing the `sent_at` claim ŌĆö so a timeout left the stage `sent_at`-locked with zero rows, and the scheduler never retried it. Three-part fix:
  1. **Batched minting** ([lib/links/mint-link.ts](../lib/links/mint-link.ts) `mintLinksBatch` + [lib/sends/kickoff.ts](../lib/sends/kickoff.ts) bulk inserts): single shared destination upsert + multi-row chunked `links`/`stage_sends` inserts. 1000-recipient kickoff ~178s ŌåÆ ~2-3s (measured).
  2. **Claim-after-materialize** ([lib/sends/scheduled.ts](../lib/sends/scheduled.ts)): `sent_at` is now stamped only AFTER kickoff succeeds; concurrency relies on the `(stage_id, contact_id)` active-send dedup index. A killed tick no longer strands the stage.
  3. **Resumable drain across ticks:** phase B (`selectDrainableStages`) drains any tracked stage with `pending` rows in budget-bounded batches, independent of `sent_at` ŌĆö large audiences send across multiple `*/15` ticks instead of one 300s window.
  - Tests: rewrote `test-scheduled-budget`/`test-scheduled-run` to the two-phase counters; added `test-scheduled-e2e` (two-tick materializeŌåÆdrainŌåÆresume, rolled back) and `verify-batch-kickoff` (timing/correctness); made `verify-send` fixture self-contained. ŌĆö Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).

## 2026-06-11 ŌĆö Fix: manual "mark sent" silently cancelled scheduled tracked sends
- `campaign_stages.sent_at` is the `send-scheduled` cron's fire-lock (`selectDueScheduledStages` requires `sent_at IS NULL`). The legacy status action ([ŌĆ”/stages/[stageId]/status/route.ts](../app/api/campaigns/[campaignId]/stages/[stageId]/status/route.ts)) stamped the same column when moving a stage to `'sent'`, so marking a tracked stage `'sent'` before its scheduled time stamped `sent_at` and permanently removed it from the scheduler ŌĆö and reverting the status didn't clear it (real incident: a 3 PM ET send never fired). Now the status route **rejects `'sent'` for tracked campaigns** (409 `tracked_stage_sent_is_pipeline_owned`), and the stage status dropdown hides `'sent'` for tracked campaigns ([campaigns/[id]/page.tsx](../app/(protected)/campaigns/[id]/page.tsx)). Added read-only `scripts/diagnose-scheduled.ts`. ŌĆö Docs updated: [04-features/sms-send-pipeline.md](04-features/sms-send-pipeline.md), [07-conventions.md](07-conventions.md).
